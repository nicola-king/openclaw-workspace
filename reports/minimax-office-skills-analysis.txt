【核心洞察】
═══════════════════════════════════════════════════

「能打开，不代表能交付」

这是被 99% AI 文档生成工具忽视的关键区别！

═══════════════════════════════════════════════════
【问题场景】AI 生成 Excel 的隐形炸弹
═══════════════════════════════════════════════════

典型失败案例:

你用 AI 生成了一份 Excel 报表
  ↓
打开文件：整整齐齐 ✅
列头对齐 ✅
公式完整 ✅
图表精美 ✅
  ↓
发给客户，客户打开
  ↓
数据透视表消失了 ❌
没有报错 ❌
没有提示 ❌
就像从来没存在过 ❌

═══════════════════════════════════════════════════
【根本原因】常用库的静默损坏
═══════════════════════════════════════════════════

openpyxl 问题:
  - 读取→写入流程
  - 静默丢失高级功能
  - 数据透视表 ❌
  - 迷你图 ❌
  - VBA 宏 ❌
  - 不报错！不警告！

python-docx 问题:
  - 嵌套表格 ❌
  - 多级目录 ❌
  - 页眉页脚精确控制 ❌
  - 修订记录 ❌
  - 结构性损坏 ❌

═══════════════════════════════════════════════════
【MiniMax 解决方案】
═══════════════════════════════════════════════════

核心原则:
「难的不是生成文件。难的是生成一份能交付的文件。」

技术选择逻辑:
  格式    常见方案          MiniMax 方案        取舍
  ─────────────────────────────────────────────────────
  Word    python-docx      .NET OpenXML SDK   部署复杂→结构完整
  Excel   openpyxl         直接操作 XML       跳过库→保高级功能
  PDF     单引擎           HTML+ReportLab     双引擎→封面 + 正文
  PPT     内容先行         约束系统先行       先定义→再生成

═══════════════════════════════════════════════════
【Word 方案】.NET OpenXML SDK
═══════════════════════════════════════════════════

选择理由:
  OpenXML SDK = 微软官方 Office 文档底层库
  对 ECMA-376 标准最完整实现
  就像用原厂零件修车，不是通用替代品

核心场景:
  1. 从零生成完整文档
  2. 不破坏格式编辑已有文档
  3. 应用设计模板 + 结构合规检查

配套资源:
  - OpenXML 格式参考文档
  - CJK 排版规范
  - 10+ 可运行代码示例

代价:
  需要部署 .NET 运行时
  比纯 Python 方案复杂

判断:
  文档质量 > 部署便利

═══════════════════════════════════════════════════
【Excel 方案】直接操作 XML
═══════════════════════════════════════════════════

核心洞察:
.xlsx 文件本质 = zip 压缩包 + XML 文件

操作方法:
  1. 解压 .xlsx 文件
  2. 定向修改目标单元格 XML 节点
  3. 重新打包

优势:
  ✅ 每次编辑只碰需要改的部分
  ✅ 样式、图表、宏保持原样
  ✅ 不会静默丢失高级功能

硬规则:
  所有衍生值必须保持真实 Excel 公式
  SUM(B2:B9) 而不是预先算好的数字 42
  用户打开文件，数据仍然联动

工具链:
  13 个独立 Python 工具脚本
  - 解压打包
  - 列插入
  - 行偏移
  - 公式校验
  - 动态重算
  - 格式审查

配套规范:
  34000 字财务格式规范
  对齐投行级数字格式和布局标准

═══════════════════════════════════════════════════
【PDF 方案】双引擎架构
═══════════════════════════════════════════════════

核心矛盾:
  封面要好看 → 需要渐变、网格、混合模式、自定义字体
  正文要稳定 → 需要分页、段落流、页眉页脚精确控制
  同一引擎很难同时做好

解决方案:
  封面：HTML + CSS → Playwright 渲染成 PDF
  正文：ReportLab → 工程级稳定控制
  合并：脚本拼接成完整 PDF

比喻:
  就像餐厅把甜品区和主厨区分开
  各自用最合适的设备和工艺
  最后由服务员端上同一张桌子

代价:
  系统复杂度提升
  两端都没有妥协

═══════════════════════════════════════════════════
【PPT 方案】约束系统先行
═══════════════════════════════════════════════════

核心洞察:
  PPT 生成里，内容从来不是最难的部分
  难的是视觉一致性

问题表现:
  字号差 1pt ❌
  间距松 2px ❌
  圆角半径不统一 ❌
  某张幻灯片颜色偏一点点 ❌
  整套 PPT 看起来像拼凑的 ❌

解决方案: 反转顺序

传统路径:
  生成内容 → 调整样式 → 修修补补

MiniMax 路径:
  定义约束系统 → 在约束里生成内容

约束系统:
  5 种标准页面类型:
    1. 封面
    2. 目录
    3. 章节分隔页
    4. 内容页
    5. 总结页
    每种都有明确布局规则和元素定位

  4 种风格配方:
    1. Sharp
    2. Soft
    3. Rounded
    4. Pill
    每个配方是完整数值集:
      - 圆角半径
      - 阴影参数
      - 边框粗细
      - 间距比例
    全部写定

换配方 = 换制服
  不是一件一件改
  而是整套替换
  视觉风格统一切换

实现:
  PptxGenJS
  编辑模板时同 Excel 策略:
    解压 .pptx → 修改 XML → 重新打包
    尽量保留原有结构

═══════════════════════════════════════════════════
【核心方法论】评估体系
═══════════════════════════════════════════════════

核心问题:
  建一个能生成文件的工具不难
  难的是知道下一个版本是不是真的更好
  而不是只是换了一种方式出错

MiniMax 方案: 三阶段循环

  执行 → 评估 → 修复
   ↑              ↓
   └──────────────┘

执行:
  跑真实用例

评估:
  用规则检查输出
  把失败提炼成可执行的问题

修复:
  送入下一轮迭代
  每次修复后立刻验证
  这次真的改好了？
  还是在别处出了新问题？

「通过」的定义:
  ❌ 不只是「文件能打开」
  ✅ 结构是否完整
  ✅ 公式是否还是公式
  ✅ 布局在读写过程中有没有静默变形
  ✅ 模板约束是否被保留

失败案例:
  保存成功但丢失数据透视表的 .xlsx → 失败
  目录能渲染但无法更新的 .docx → 失败

【技术选择总结】
═══════════════════════════════════════════════════

格式    放弃什么            得到什么
─────────────────────────────────────────────
Word    简单部署           结构完整性
Excel   库的便利性         高级功能不丢失
PDF     系统简洁性         封面表达力 + 正文稳定性
PPT     生成灵活性         视觉一致性

共同点:
  每一个选择，都是用更高的系统复杂度
  换更可靠的最终输出

核心判断:
  代价和选择总是成对出现的
  他们只是把这件事说清楚
  然后选了那个更难的选项

【对太一系统的启示】
═══════════════════════════════════════════════════

当前太一文档生成能力:
  ✅ md2pdf.sh - Markdown 转 PDF
  ✅ 基础 Excel 生成
  ✅ 基础 Word 生成
  ❌ 高级功能支持不足
  ❌ 评估体系缺失

差距分析:

1. Excel 高级功能
   现状：使用 openpyxl 基础功能
   差距：数据透视表、图表、宏会静默丢失
   建议：学习 MiniMax XML 直接操作方案

2. Word 结构完整性
   现状：未部署专业库
   差距：嵌套表格、多级目录支持弱
   建议：评估 .NET OpenXML SDK 可行性

3. PDF 设计系统
   现状：单引擎（Chrome headless）
   差距：封面设计表达力有限
   建议：考虑双引擎架构（HTML+ReportLab）

4. 评估体系
   现状：文件生成即完成
   差距：无「能交付」标准检查
   建议：构建三阶段循环（执行→评估→修复）
