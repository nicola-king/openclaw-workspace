# 漏洞封堵模式

> **漏洞即必然：Agent 在压力下必找漏洞 → 必须明确禁止具体变通方式**

---

## 🎯 核心原则

**铁律：**
> Violating the letter of the rules is violating the spirit of the rules.

**洞察：**
> Agent 在压力下会创造性违规 → 必须明确禁止具体变通方式

---

## 📋 漏洞封堵四件套

### 1. 铁律声明

```markdown
## 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

写代码前没测试？删除。重来。
```

**要点：**
- 一句话核心规则
- 全部大写强调
- 违反后果明确

---

### 2. 无例外列表

```markdown
## 无例外

**不因为任何理由变通：**
- 不保留作"参考"
- 不"边写边看"
- 不"先实现再补测试"
- 不"这次情况特殊"
- 不"我已经手动测过了"
- 不"测试只是形式"
- 不"时间来不及"
- 不"这个功能不重要"

**删除就是删除。**
```

**要点：**
- 至少 5-8 条具体变通方式
- 用 Agent 的原话（verbatim）
- 每条都是实际出现过的借口

---

### 3. 合理化表格

```markdown
## 合理化表格

| 借口 | 现实 |
|------|------|
| "太简单不用测试" | 简单代码也会错，测试 30 秒 |
| "我待会再测" | 事后测试 = "这代码干嘛？" vs "应该干嘛？" |
| "精神比形式重要" | 违反字面规则 = 违反精神 |
| "这次不一样" | 每次都觉得不一样，所以每次都错 |
| "我已经手动测过了" | 手动测试不可重复，不是测试 |
| "客户急着要" | 客户更急着要正确的东西 |
| "测试写不出来" | 测试写不出来说明需求不清楚 |
| "老板说不用测" | 老板不为你的错误买单 |
```

**来源：** 基线测试中 Agent 的 verbatim 借口。

**要点：**
- 左栏：Agent 原话（引号内）
- 右栏：一句话反驳
- 至少 8 条

---

### 4. 红旗列表

```markdown
## 红旗列表 - STOP and Start Over

**出现以下任何一条，立即删除代码，重来：**

- [ ] 代码写在前，测试在后
- [ ] "我手动测过了"
- [ ] "测试只是形式"
- [ ] "这次情况特殊"
- [ ] "我先实现再补测试"
- [ ] "这个功能不重要"
- [ ] "时间来不及了"

**所有这些都意味着：你在 rationalize，不是在执行。**
```

**要点：**
- 可操作的检查项
- 自我检测触发器
- 最后一句点明本质

---

## 🔍 漏洞来源

### 来源 1：基线测试

**方法：**
```
1. 无技能跑压力场景
2. 记录 Agent verbatim 行为
3. 提取合理化借口
4. 写入技能对应禁止
```

**示例：**
```
压力："这个很急，5 分钟内搞定"
Agent："我先快速实现，再补测试"
→ 写入无例外："不'先实现再补测试'"
```

### 来源 2：生产违规

**方法：**
```
1. 监控技能使用情况
2. 发现创造性违规
3. 更新技能堵漏洞
4. 重新测试验证
```

**示例：**
```
违规：Agent 保留代码作"参考"
→ 写入无例外："不保留作'参考'"
```

### 来源 3：压力组合

**常见压力组合：**
| 组合 | 示例 | 典型违规 |
|------|------|---------|
| 时间 + 沉没成本 | "我写了一半了，很急" | "先看完再补测试" |
| 权威 + 复杂度 | "老板说这个太复杂不用测" | "这次情况特殊" |
| 疲劳 + 时间 | "改了 10 次了，差不多行了" | "手动测过了" |

## 🚨 常见漏洞类型

### 类型 1：精神>形式论

**借口：** "重要的是精神，不是形式"
**封堵：** "违反字面规则 = 违反精神"

### 类型 2：例外论

**借口：** "这次情况特殊"
**封堵：** "每次都觉得不一样，所以每次都错"

### 类型 3：延迟论

**借口：** "我待会再补"
**封堵：** "事后测试 = '这代码干嘛？' vs '应该干嘛？'"

### 类型 4：参考论

**借口：** "我保留作参考，不直接用"
**封堵：** "不保留作'参考'，删除就是删除"

### 类型 5：手动测试论

**借口：** "我已经手动测过了"
**封堵：** "手动测试不可重复，不是测试"

### 类型 6：权威论

**借口：** "我老板/客户说不用测"
**封堵：** "老板/客户不为你的错误买单"

---

## 💡 核心洞察

### 1. 漏洞不是意外，是必然
> Agent 在压力下必找漏洞 → 预期它，准备它，封堵它

### 2. verbatim 是最强武器
> 用 Agent 的原话写封堵 → 它无法说"我没说过"

### 3. 反驳要一句话
> 长反驳会被忽略 → 一句话击中要害

### 4. 迭代是常态
> 第一轮必漏 → 发现→封堵→重测 → 3-5 轮收敛
