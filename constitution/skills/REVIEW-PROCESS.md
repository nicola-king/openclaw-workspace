# 三级审查流程执行手册

## 审查级别速查表

| 变更类型 | 审查级别 | 批准人 | 流程时间 |
|----------|----------|--------|----------|
| 宪法修订 | L3 | SAYELF | 需预约 |
| 核心技能变更 | L3 | SAYELF | 需预约 |
| 新 Bot 接入 | L3 | SAYELF | 需预约 |
| 跨域技能 | L2 | 太一 + 告知 | 10 分钟 |
| 新功能模块 | L2 | 太一 + 告知 | 10 分钟 |
| 工具技能更新 | L1 | 太一 | 即时 |
| 文档修正 | L1 | 太一 | 即时 |

---

## L1 快速审查流程

### 触发条件

- [ ] 修改 L2 工具技能文件
- [ ] 修正文档错别字
- [ ] 调整配置参数
- [ ] 添加注释/说明

### 执行步骤

```bash
# 1. 内部标注
[审查：L1] TASK-$(date +%Y%m%d)-$(seq)

# 2. 执行修改
edit constitution/skills/XXX.md

# 3. 写入 memory 记录
追加到 memory/YYYY-MM-DD.md

# 4. 更新 HEARTBEAT.md 检查清单
```

### Memory 记录模板

```markdown
【TASK-20260322-XXX】[审查：L1] [能力涌现]
- 变更：更新 XXX.md 添加 YYY 功能
- 原因：ZZZ 需求
- 影响：无风险
- 回滚：还原文件即可
```

### 示例

```markdown
【TASK-20260322-006】[审查：L1] [能力涌现]
- 变更：创建 SKILL-MARKET.md 定义双层技能市场
- 原因：SAYELF 批准 P1 执行
- 影响：L2 技能可快速迭代，L1 技能受保护
- 回滚：删除文件即可
```

## L2 专项审查流程

### 触发条件

- [ ] 开发跨域技能（涉及 2+ Bot）
- [ ] 添加新功能模块
- [ ] 集成外部 API
- [ ] 变更数据结构

### 执行步骤

```
1. 太一发起 → 编写风险评估报告
2. SAYELF 告知 → 等待 10 分钟确认
3. SAYELF 确认/否决 → 执行或修改
4. 写入 memory 记录
5. 更新 HEARTBEAT.md
```

### 风险评估报告模板

```markdown
【L2 审查申请 · TASK-XXXXXXX-XXX】

## 变更内容
[详细描述要做什么]

## 变更原因
[为什么需要这个变更]

## 影响评估
- 影响范围：[哪些 Bot/模块会受影响]
- 正面影响：[预期收益]
- 负面影响：[潜在风险]

## 风险评估
- 回滚难度：[低/中/高] + 说明
- 数据风险：[无/低/中/高] + 说明
- 依赖变更：[无/有] + 说明

## 测试计划
[如何验证变更成功]

## 建议决策
- [ ] 批准执行
- [ ] 需要修改（说明：___）
- [ ] 升级为 L3 审查

---
*申请人：太一 | 日期：YYYY-MM-DD HH:mm*
```

### SAYELF 回复选项

| 回复 | 含义 |
|------|------|
| `批准` / `同意` / `执行` | L2 审查通过，继续执行 |
| `需要修改` + 说明 | 按说明修改后重新提交 |
| `升级为 L3` | 需要更详细方案 |
| `拒绝` | 终止变更 |

### Memory 记录模板

```markdown
【TASK-20260322-XXX】[审查：L2] [决策]
- 变更：[简述]
- SAYELF 回复：[批准/修改/拒绝]
- 执行状态：[已完成/待执行/已终止]
- 结果：[成功/失败/部分成功]
```

## L3 深度审查流程

### 触发条件

- [ ] 修改宪法文件（L1 技能）
- [ ] 变更核心技能（MULTI-BOT 等）
- [ ] 接入新 Bot
- [ ] 架构调整
- [ ] 安全边界变更

### 执行步骤

```
1. 太一发起 → 编写完整方案
2. SAYELF 审查 → 提问/修改/批准
3. SAYELF 书面批准 → 获得明确"批准"指令
4. 执行变更 → 按方案实施
5. 验证结果 → 测试确认
6. 写入 memory + 更新文档
```

### SAYELF 审查要点

**SAYELF 审查时应关注：**

1. **变更必要性** - 是否真的需要？
2. **方案完整性** - 是否考虑周全？
3. **风险可控性** - 最坏情况能否接受？
4. **回滚可行性** - 出错能否快速恢复？
5. **长期影响** - 对未来架构的影响？

### SAYELF 回复格式

```markdown
【L3 审查批复】

**TASK 编号：** TASK-YYYYMMDD-XXX
**批复人：** SAYELF
**日期：** YYYY-MM-DD HH:mm
```

## 紧急情况处理

### 紧急跳过审查

**适用场景：**
- 系统故障需立即修复
- 安全漏洞需立即修补
- SAYELF 无法及时响应但情况紧急

**流程：**
```
太一判断紧急 → 执行变更 → 
事后 24 小时内补 L3 审查 → 
写入 memory 说明紧急原因
```

### 事后补审模板

```markdown
【紧急变更 · 事后补审】

**TASK 编号：** TASK-YYYYMMDD-XXX
**变更时间：** YYYY-MM-DD HH:mm
**补审时间：** YYYY-MM-DD HH:mm
**紧急原因：** [为什么无法等待正常审查]

## 变更内容
[做了什么]

## 紧急原因说明
[为什么跳过审查]

## 结果验证
[变更后结果如何]

## 请求追认
- [ ] 追认批准
- [ ] 需要修改
- [ ] 拒绝并要求回滚

---
*申请人：太一 | 日期：YYYY-MM-DD*
```

### 审查流程优化

**优化触发条件：**
- 同一问题重复出现 3 次以上
- SAYELF 提出优化建议
- 太一自检发现瓶颈

**优化流程：**
```
问题识别 → 方案提出 → L2 审查 → 
执行优化 → 验证效果 → 写入 memory
```
