# A-Z 分发机制（任务防重复认领）

## 核心原则

**负熵法则：** 减少任务分配混乱，避免重复劳动

**价值基石：** 专业分工，高效协作

**简化优先：** 26 字母编号，简单易记

## 任务认领流程

### 流程步骤

```
1. 太一发布任务 → 分配字母编号
2. Bot 认领 → 确认职责域匹配
3. 执行任务 → 写入 HEARTBEAT.md
4. 完成报告 → 更新状态为✅
```

### 认领规则

**规则 1：职责域优先**
- Bot 只能认领自己字母范围内的任务
- 跨域任务需太一批准

**规则 2：先到先得**
- 同一任务多个 Bot 可申请 → 先认领者得
- 太一有最终分配权

**规则 3：容量限制**
- 每 Bot 同时执行任务≤3 个
- 超出需完成现有任务或转交

**规则 4：超时释放**
- 认领后 24h 无进展 → 自动释放
- 太一可重新分配

## 任务状态管理

### 状态定义

| 状态 | 符号 | 说明 |
|------|------|------|
| 待认领 | 🟡 | 任务已发布，等待 Bot 认领 |
| 已认领 | 🔵 | Bot 已认领，执行中 |
| 已完成 | ✅ | 任务完成，结果验收通过 |
| 已释放 | ⚪ | 超时/转交，重新开放认领 |
| 已取消 | ❌ | 任务取消，无需执行 |

## 冲突解决

### 冲突类型

| 类型 | 场景 | 解决方案 |
|------|------|----------|
| 职责域重叠 | 任务属于多 Bot 职责 | 太一指定或 Bot 协商 |
| 同时认领 | 多 Bot 同时认领 | 先提交者得 |
| 能力不足 | Bot 无法完成任务 | 转交或请求协助 |
| 优先级冲突 | 多任务高优先级 | 太一调整优先级 |

### 转交流程

```
Bot A 申请转交 → 太一审核 → 
发布认领通知 → Bot B 认领 → 
更新 HEARTBEAT.md → 继续执行
```
