# 极简 SDLC 流程（简化版）

## 核心原则

**不为了复杂而复杂** - 仅保留 4 个核心阶段

```
规划 → 实现 → 审查 → 记录
```

**省略：** 测试框架、自动化部署、监控告警（由太一人工替代）

---

## 四阶段流程

### 阶段 1：规划（Plan）

**输入：** SAYELF 需求或太一自检发现

**输出：** HEARTBEAT.md 待办事项

**步骤：**
1. 识别任务类型 → 确定审查级别（L1/L2/L3）
2. 分配 TASK 编号 → `TASK-YYYYMMDD-NNN`
3. 写入 HEARTBEAT.md → 待办事项表格
4. 如需 L2/L3 → 编写审查申请

**模板：**
```markdown
## 待办事项

| 编号 | 任务 | 状态 | 优先级 | 审查级别 |
|------|------|------|--------|----------|
| `TASK-20260322-009` | [任务描述] | 🟡 规划中 | P2 | L1/L2/L3 |
```

---

### 阶段 2：实现（Build）

**输入：** HEARTBEAT.md 待办事项

**输出：** 技能文件/文档/配置

**步骤：**
1. 读取 TASK 详情 → 理解需求
2. 创建/修改文件 → 执行实现
3. 内部自检 → 确保无低级错误
4. 准备审查 → 根据级别提交

**自检清单：**
- [ ] 文件格式正确（SKILL.md 遵循规范）
- [ ] 无语法错误
- [ ] 无硬编码敏感信息
- [ ] 文档更新完整
- [ ] 与现有技能无冲突

---

### 阶段 3：审查（Review）

**输入：** 实现完成的文件

**输出：** 审查通过/修改/拒绝

**步骤：**
1. 判断审查级别 → L1/L2/L3
2. 执行对应审查流程
3. 获得批准 → 继续或修改
4. 记录审查结果

**审查级别判断：**
```
修改宪法？ → L3
新 Bot？ → L3
跨域？ → L2
新功能？ → L2
工具更新？ → L1
文档修正？ → L1
```

---

### 阶段 4：记录（Record）

**输入：** 审查通过的变更

**输出：** memory/YYYY-MM-DD.md 记录

**步骤：**
1. 写入 memory 文件 → 追加模式
2. 标注类型 → [决策] [任务] [洞察] [能力涌现] [宪法修订]
3. 更新 HEARTBEAT.md → 标记完成
4. 可选：更新文档索引

**模板：**
```markdown
【TASK-20260322-XXX】[审查：L1/L2/L3] [类型]
- 变更：[简述]
- 原因：[为什么]
- 结果：[成功/失败]
- 影响：[正面/负面]
- 回滚：[如何撤销]
```

## 阶段对比（完整版 vs 简化版）

| 阶段 | 完整 SDLC | 太一简化版 | 说明 |
|------|-----------|------------|------|
| **需求** | 需求分析文档 | HEARTBEAT.md 一行 | 简化 90% |
| **规划** | 详细项目计划 | TASK 编号 + 审查级别 | 简化 80% |
| **设计** | 架构设计文档 | 太一内部思考 | 省略文档 |
| **实现** | 编码 + 单元测试 | 编码 + 自检清单 | 省略自动化测试 |
| **审查** | Code Review + 审批 | L1/L2/L3审查 | 保留核心 |
| **测试** | 集成测试 + E2E | 太一人工验证 | 简化 95% |
| **部署** | CI/CD流水线 | 直接写入文件 | 简化 90% |
| **监控** | 告警 + 指标 | SAYELF 反馈 | 简化 100% |

**简化率：** 约 85-90%

## 时间节奏整合

### 心跳检查时间表

| 时间 | 检查内容 | 输出 |
|------|----------|------|
| **09:00** | 晨间检查 | 待办 + 日历 + 天气 |
| **17:00** | 晚间检查 | 进度 + 明日计划 |
| **23:00** | 夜间压缩 | Session 结束协议 |

### 晨间检查（09:00）

**检查清单：**
- [ ] 读取 HEARTBEAT.md → 恢复上下文
- [ ] 检查日历 → 今日事件
- [ ] 检查天气 → 如有外出计划
- [ ] 优先级排序 → 调整待办顺序

**输出：**
```markdown
【晨间检查 · YYYY-MM-DD 09:00】
- 今日事件：[列表]
- 优先级任务：[TOP3]
- 天气：[如有需要]
- 备注：[其他]
```

### 晚间检查（17:00）

**检查清单：**
- [ ] 检查待办完成状态
- [ ] 更新 HEARTBEAT.md
- [ ] 规划明日任务
- [ ] 如有未完成 → 说明原因

**输出：**
```markdown
【晚间检查 · YYYY-MM-DD 17:00】
- 今日完成：[列表]
- 未完成：[列表 + 原因]
- 明日计划：[列表]
- 需要 SAYELF 决策：[如有]
```

### 夜间压缩（23:00）

**执行 Session 结束协议：**
1. 读取当日 memory → 提取关键决策
2. 压缩上下文 → 写入 memory
3. 更新 HEARTBEAT.md → 未完成事项
4. 生成日报框架（如到时间点）

**输出：**
```markdown
【Session 压缩 · YYYY-MM-DD 23:00】
关键决策：
1. 
2. 

待办事项：
- [ ] 

上下文已压缩，下次 Session 加载本记录恢复连续性。
```

## 与多 Bot 协作整合

### Bot 职责映射

| SDLC 阶段 | 负责 Bot | 说明 |
|-----------|---------|------|
| **规划** | 太一 + 罔两 | 罔两提供数据分析 |
| **实现** | 素问/山木 | 素问技术/山木内容 |
| **审查** | 太一 + 庖丁 | 庖丁评估成本 |
| **记录** | 太一 | 统一写入 memory |

### 任务分发编号（A-Z）

**防重复机制：**
```
山木-A → 内容创意任务
素问-B → 技术开发任务
罔两-C → 数据分析任务
庖丁-D → 成本评估任务
知几-E → 量化分析任务
```

**使用方式：**
```markdown
【TASK-20260322-009】[审查：L1] [能力涌现]
- 分发编号：山木-A
- 任务：创建内容创意流程
- 状态：执行中
```

## 持续改进

### 月度回顾

**每月初执行：**
1. 读取上月 memory 文件
2. 统计任务完成情况
3. 识别改进点
4. 更新 SDLC 流程（L1 审查）

**回顾模板：**
```markdown
【月度回顾 · YYYY-MM】

## 任务统计
- 总任务数：X
- 完成率：X%
- 平均耗时：X 小时

## 问题分析
- 重复问题：[列表]
- 瓶颈环节：[列表]

## 改进措施
1. [措施 1]
2. [措施 2]

## 下月目标
- [目标 1]
- [目标 2]
```

### 流程优化触发

**优化条件：**
- 同一问题重复 3 次以上
- SAYELF 提出优化建议
- 太一自检发现瓶颈

**优化流程：**
```
问题识别 → 太一分析 → 方案提出 → 
L1 审查 → 执行优化 → 验证效果 → 
写入 memory [流程优化]
```
