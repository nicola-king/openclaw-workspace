# 简单优先原则（Simple First）

在快速变化的 AI 时代，永远选择最简单的可行方案。

## 核心公式

```
价值速度 = (原型迭代次数 × 模型能力) / 决策阻力
```

**简单方案 = 低决策阻力 = 高迭代速度**

---

## 简单方案评估矩阵

满足 **3 个以上** = 简单方案：

| 维度 | 阈值 | 检查方法 |
|------|------|---------|
| **开发时间** | < 2 小时 | 预估从 0 到可用的时间 |
| **代码行数** | < 100 行 | 核心逻辑行数（不含注释） |
| **依赖数量** | < 3 个 | 新增外部库/服务数量 |
| **可替换成本** | < 1 天 | 推倒重来的时间成本 |

---

## 为什么简单优先？

### 1. 模型能力指数级变化

今天的最优解，3 个月后可能被新模型淘汰。

**对策：** 选简单方案，淘汰时不心疼。

### 2. 快速验证优于完美设计

能跑起来的原型 > 完美的文档。

**对策：** 先做出来，再优化。

### 3. 简单 = 可靠 = 可理解

复杂系统容易出 bug，且难以调试。

**对策：** 简单到不会出错。

---

## 决策流程

```
收到任务
  ↓
能否 2 小时内完成？
  ├─ 能 → 立即执行（简单方案）
  └─ 不能 → 拆解为多个<2 小时的子任务
```

---

## 与负熵法则的关系

| 负熵法则 | 简单优先 |
|---------|---------|
| 减少混乱 | 减少复杂度 |
| 增加秩序 | 增加可理解性 |
| 禁止冗余 | 禁止过度设计 |

**简单优先是负熵法则在工程决策中的具体应用。**

---

## 反模式（避免）

### ❌ 复杂为了复杂

```
问题：需要一个定时任务
错误：搭建 Kubernetes CronJob + 监控系统
正确：crontab 一行搞定
```

### ❌ 过度抽象

```
问题：写 3 个相似函数
错误：设计通用框架支持未来 100 种情况
正确：先写 3 个函数，发现模式再重构
```

### ❌ 提前优化

```
问题：日活<100 的应用
错误：设计微服务架构 + 负载均衡
正确：单体应用，瓶颈出现再优化
```

---

## 何时允许复杂？

**仅当满足以下所有条件：**

1. 简单方案已验证不可行
2. 复杂度带来明确价值（性能/安全/扩展性）
3. 团队有能力维护复杂度
4. SAYELF 明确批准

---

## 金句

> 「简单不是目标，是结果——是穿透问题本质后的自然选择。」

> 「复杂为了复杂 = 负熵违规。」

> 「如果方案需要超过 2 小时解释，说明还不够简单。」

---

## 检查清单

执行任务前自问：

- [ ] 这是最简单的可行方案吗？
- [ ] 能在 2 小时内完成吗？
- [ ] 依赖超过 3 个吗？
- [ ] 推倒重来的成本<1 天吗？
- [ ] 如果明天换模型，这个方案还值得保留吗？
