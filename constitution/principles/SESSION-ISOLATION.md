**版本：** v1.0

**级别：** 宪法级（必须遵守）  
**授权：** SAYELF

## 📜 核心原则

**「微信通道是共享入口，各 Bot 会话必须隔离」**

---

## 🎯 为什么需要隔离

### 问题场景

```
❌ 错误做法（无隔离）：
用户 → 微信 → 太一 → 知几 → 山木 → 素问...
所有 Bot 共享同一上下文 → 信息泄露/混乱

例如：
- 知几的交易信号泄露给山木
- 素问的技术日志混入太一日记
- 庖丁的财务数据暴露给罔两
```

### 风险

| 风险类型 | 说明 | 后果 |
|---------|------|------|
| **信息泄露** | Bot 间数据交叉 | 隐私/安全风险 |
| **上下文混乱** | 多 Bot 回复同一消息 | 用户体验差 |
| **职责越界** | Bot 处理非职责域任务 | 输出质量下降 |
| **Token 浪费** | 无关上下文加载 | 成本增加 |

---

## 🔒 隔离规则

### 1. 通道隔离

**微信通道 = 共享入口，不是共享上下文**

```
微信用户消息
    ↓
太一（唯一入口 Bot）
    ↓
┌─────────────────────────────────┐
│  太一判断职责域，按需调度        │
│                                 │
│  ┌───────┐ ┌───────┐ ┌───────┐ │
│  │ 知几  │ │ 山木  │ │ 素问  │ │
│  │(交易) │ │(内容) │ │(技术) │ │
│  └───────┘ └───────┘ └───────┘ │
│                                 │
│  各 Bot 独立会话，不共享上下文    │
└─────────────────────────────────┘
    ↓
太一汇总 → 微信用户
```

---

### 2. 会话隔离

**每个 Bot 有独立会话空间**

| Bot | 会话空间 | 可访问上下文 |
|-----|---------|-------------|
| **太一** | `session:taiyi:wechat` | 全部 + 用户消息 |
| **知几** | `session:zhiji:isolated` | 仅交易相关 |
| **山木** | `session:shanmu:isolated` | 仅内容相关 |
| **素问** | `session:suwen:isolated` | 仅技术相关 |
| **罔两** | `session:wangliang:isolated` | 仅数据相关 |
| **庖丁** | `session:paoding:isolated` | 仅财务相关 |

---

### 3. 消息路由

**太一作为唯一路由入口**

```
用户消息 → 太一
    ↓
太一判断：
├── 交易相关 → 调度知几 → 知几独立会话处理 → 返回太一
├── 内容相关 → 调度山木 → 山木独立会话处理 → 返回太一
├── 技术相关 → 调度素问 → 素问独立会话处理 → 返回太一
├── 数据相关 → 调度罔两 → 罔两独立会话处理 → 返回太一
├── 财务相关 → 调度庖丁 → 庖丁独立会话处理 → 返回太一
└── 综合/决策 → 太一直接处理

最终：太一汇总 → 回复用户
```

---

### 4. 上下文加载

**各 Bot 只加载自己会话的上下文**

```markdown
❌ 错误做法：
知几加载 → 太一 + 山木 + 素问 + 罔两 + 庖丁 的上下文

✅ 正确做法：
知几加载 → 仅知几会话上下文 + 必要共享记忆（MEMORY.md 公开部分）
```

**共享记忆分级：**

| 记忆类型 | 内容 | 访问权限 |
|---------|------|---------|
| **MEMORY.md (公开)** | 系统配置/公共知识 | 所有 Bot 可访问 |
| **MEMORY.md (私有)** | 用户隐私/财务数据 | 仅太一可访问 |
| **会话记忆** | 对话历史/临时上下文 | 仅本 Bot 可访问 |
| **每日记忆** | `memory/YYYY-MM-DD.md` | 按职责域访问 |

---

### 5. 响应合并

**多 Bot 响应由太一统一合并**

```
场景：用户问"今天交易和内容情况如何？"

太一分解：
├── 交易情况 → 调度知几 → 知几返回交易报告
└── 内容情况 → 调度山木 → 山木返回内容报告

太一合并：
┌─────────────────────────────────┐
│ 📊 今日汇总                      │
│                                 │
│ 【交易 · 知几】                 │
│ - 气象数据：189 条              │
│ - 交易机会：0 个                │
│                                 │
│ 【内容 · 山木】                 │
│ - 小红书：已发布壁纸            │
│ - 热点：治愈系主题上升          │
└─────────────────────────────────┘
    ↓
一条消息回复用户
```

## 🛡️ 安全边界

### 禁止行为

| 行为 | 状态 | 原因 |
|------|------|------|
| Bot 直接回复微信用户 | ❌ 禁止 | 破坏统一入口 |
| Bot 访问其他 Bot 会话 | ❌ 禁止 | 信息泄露风险 |
| Bot 加载全部上下文 | ❌ 禁止 | Token 浪费 + 混乱 |
| Bot 越权处理任务 | ❌ 禁止 | 职责域混乱 |
| 多条消息分开发送 | ❌ 禁止 | 用户体验差 |

### 必须行为

| 行为 | 状态 | 原因 |
|------|------|------|
| 太一作为唯一入口 | ✅ 必须 | 统一路由 |
| 会话独立隔离 | ✅ 必须 | 安全/隐私 |
| 上下文按需加载 | ✅ 必须 | 效率/质量 |
| 响应统一合并 | ✅ 必须 | 用户体验 |
| 职责域明确划分 | ✅ 必须 | 专业输出 |

## 📋 技术实现

### 1. 会话管理

```python
# 太一调度示例
def dispatch_task(bot, task, user_message):
    # 创建/获取 Bot 独立会话
    session_key = f"session:{bot}:isolated"
    
    # 仅传递任务相关上下文
    context = {
        "task": task,
        "user_message": user_message,
        "shared_memory": load_public_memory(),  # 仅公开部分
        # 不传递其他 Bot 会话内容
    }
    
    # Bot 独立处理
    result = bot.execute(context)
    
    # 返回太一汇总
    return result
```

### 2. 上下文过滤

```python
# 加载上下文时过滤
def load_context(bot_name):
    # 本 Bot 会话历史
    session_history = load_session_history(bot_name)
    
    # 公开记忆（所有 Bot 可访问）
    public_memory = load_memory("MEMORY.md", section="public")
    
    # 职责域记忆
    domain_memory = load_memory(f"memory/{bot_name}/")
    
    # 合并（不包含其他 Bot 私有内容）
    return session_history + public_memory + domain_memory
```

### 3. 消息合并

```python
# 太一汇总多 Bot 响应
def merge_responses(responses):
    merged = "📊 今日汇总\n\n"
    
    for bot, response in responses.items():
        merged += f"【{bot}】\n{response}\n\n"
    
    # 一条消息发送
    send_wechat_message(merged)
```

## 📝 违规案例

### 案例 1：信息泄露

```
❌ 错误：
用户问知几："今天交易如何？"
知几回复时加载了庖丁的财务数据
→ 财务数据泄露给交易 Bot

✅ 正确：
知几仅加载交易相关上下文
财务数据由太一按需提供给庖丁
```

### 案例 2：上下文混乱

```
❌ 错误：
山木加载了知几的交易历史 + 素问的技术日志
→ 生成内容时混入交易/技术术语
→ 用户收到混乱消息

✅ 正确：
山木仅加载内容创作相关上下文
输出纯净内容报告
```

### 案例 3：多条消息轰炸

```
❌ 错误：
太一 → 发送交易报告
知几 → 发送交易详情
山木 → 发送内容报告
素问 → 发送技术日志
→ 用户收到 4 条消息

✅ 正确：
太一汇总所有 Bot 报告
→ 用户收到 1 条完整消息
```
