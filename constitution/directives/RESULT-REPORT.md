## 🎯 核心原则

### 第一原则：完成必有汇报

> 任何任务完成，必须生成成果汇报 —— 无汇报 = 未完成

**要求**：
1. 任务完成后 5 分钟内生成汇报
2. 汇报必须包含可验证成果
3. 汇报必须发送给 SAYELF
4. 汇报必须归档到 memory

**违宪行为**：
- ❌ 任务完成不汇报
- ❌ 汇报延迟 >5 分钟
- ❌ 汇报无实质内容
- ❌ 汇报无文件交付

### 各类型汇报特殊要求

#### 调研报告
必须包含：
- 调研对象清单（≥3 个）
- 核心发现（≥5 条）
- 数据对比表
- 可执行建议（≥3 条）

#### 策略方案
必须包含：
- 对标分析
- 差异化定位
- 投入产出预测（12 个月）
- 执行时间表（分季度）
- 风险管控措施

#### 代码开发
必须包含：
- 功能清单
- 测试结果（通过率）
- 部署说明
- 回滚方案

#### 故障修复
必须包含：
- 根因分析（5Why）
- 修复动作
- 验证结果
- 预防措施

## 🚀 汇报生成流程

### 标准流程

```
1. 任务完成 → 触发汇报生成
2. 选择汇报模板 → 根据任务类型
3. 收集成果数据 → 文件/数据/指标
4. 生成汇报内容 → 按模板填充
5. 发送 SAYELF → 微信 + 文件
6. 归档到 memory → memory/YYYY-MM-DD.md
7. 更新任务状态 → HEARTBEAT.md
```

### 自动化程度

| 阶段 | 自动化 | 人工干预 |
|------|--------|---------|
| 成果收集 | ✅ 100% 自动 | 无 |
| 汇报生成 | ✅ 100% 自动 | 无 |
| 发送渠道 | ✅ 100% 自动 | 无 |
| 归档 memory | ✅ 100% 自动 | 无 |
| 质量检查 | 🟡 90% 自动 | 异常时人工 |

## 📈 汇报质量门禁

### 必须通过的检查

| 检查项 | 标准 | 未通过处理 |
|--------|------|-----------|
| **成果量化** | 必须有数字指标 | 补充数据 |
| **文件交付** | 必须有文件路径/链接 | 补充文件 |
| **可验证性** | 成果可被第三方验证 | 补充证明 |
| **行动建议** | 必须有下一步建议 | 补充建议 |
| **格式规范** | 符合模板要求 | 重新生成 |

### 质量评分

| 等级 | 分数 | 要求 |
|------|------|------|
| **S 级** | 95-100 | 超出预期，有额外洞察 |
| **A 级** | 85-94 | 完整规范，有量化成果 |
| **B 级** | 70-84 | 基本完整，缺少部分数据 |
| **C 级** | <70 | 不合格，需重新生成 |

**目标**: 90% 汇报达到 A 级以上

## 🔧 技能集成

### Result-Reporter Skill

**位置**: `skills/result-reporter/`

**核心 API**:
```python
from result_reporter import ResultReporter, ReportTemplate

reporter = ResultReporter()

# 选择模板
template = reporter.select_template(task_type)

# 生成汇报
report = reporter.generate(template, task_data)

# 发送汇报
await reporter.send(report, channel="wechat")

# 归档
reporter.archive(report, memory_file)
```

**模板库**:
- `templates/research-report.md` - 调研报告
- `templates/strategy-plan.md` - 策略方案
- `templates/dev-report.md` - 代码开发
- `templates/fix-report.md` - 故障修复
- `templates/content-report.md` - 内容创作

## 🔐 安全基线

### 禁止行为

- ❌ 汇报中包含虚假数据

- ❌ 汇报中泄露敏感信息
- ❌ 汇报后不归档

### 必须行为

- ✅ 汇报必须真实准确
- ✅ 失败必须如实汇报
- ✅ 敏感信息必须脱敏
- ✅ 汇报必须及时归档
- ✅ 汇报必须可追溯
