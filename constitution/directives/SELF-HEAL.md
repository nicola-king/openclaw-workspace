## 🎯 核心原则

### 第一原则：自愈必须智能

> 自愈不是盲目重启，而是智能诊断 + 精准修复 + 效果验证

**要求**：
1. 每次自愈前必须诊断根因
2. 自愈动作必须有明确目标
3. 自愈后必须验证效果
4. 失败必须记录并分析

---

## 🛡️ 防死循环法则

### 铁律：永不无限重试

**三层保护机制**：

| 保护 | 阈值 | 触发后 |
|------|------|--------|
| 单问题自愈次数 | **3 次** | 该问题停止自愈 |
| 总失败次数 | **5 次** | 全系统停止自愈 |
| 自愈冷却时间 | **10 分钟** | 间隔内不自愈 |

**违宪行为**：
- ❌ 无限制重试同一问题
- ❌ 忽略失败计数继续自愈
- ❌ 绕过冷却时间强制自愈
- ❌ 不记录自愈历史

---

## 🚨 人工干预触发

### 触发条件（满足任一即触发）

1. **单问题自愈 ≥ 3 次** 仍未解决
2. **总失败次数 ≥ 5 次**
3. **连续失败 ≥ 3 次**
4. **发现未知问题类型**

### 触发后动作

```
1. 🚨 状态变为 intervention_required
2. 📢 发送告警到微信（SAYELF）
3. ⏸️ 停止所有自愈尝试
4. 📝 记录干预原因和时间
5. ⏳ 等待人工修复
```

### 人工修复后

SAYELF 发送 `/自愈重置` 后：
- ✅ 清除自愈计数
- ✅ 重置失败统计
- ✅ 恢复自愈功能
- ✅ 记录干预结束时间

## 📝 成果汇报

### 汇报类型

**1. 单次成果汇报**（每次自愈完成）

必须包含：
- ✅/❌ 自愈结果
- 问题 ID 和名称
- 自愈动作
- 尝试次数
- 详细信息（耗时、PID 等）
- 统计数据（连续成功、总失败）
- 人工干预警告（如触发）

**2. 周期汇报**（每 10 分钟）

必须包含：
- 系统状态
- 本次周期统计（成功/失败次数）
- 最近自愈记录（最多 5 条）
- 防死循环保护状态
- 人工干预警告（如触发）

### 汇报渠道

- 主渠道：微信（taiyi 账号 → SAYELF）
- 备用渠道：Telegram（如微信故障）

## 🔧 自愈流程

### 标准流程

```
1. 发现问题 → 记录到 currentIssues
2. 检查 can_heal() → 确认可以自愈
3. 开始自愈 → start_heal(issue_id)
4. 执行自愈动作 → 诊断 + 修复 + 验证
5. 记录结果 → success/failure
6. 生成汇报 → 发送成果汇报
7. 检查干预 → needs_intervention()
```

### 自愈动作分类

| 级别 | 动作 | 风险 | 审批 |
|------|------|------|------|
| L1 | 重启服务 | 低 | 自动 |
| L2 | 清理缓存/临时文件 | 低 | 自动 |
| L3 | 重置配置 | 中 | 自动 + 汇报 |
| L4 | 重装组件 | 高 | 需人工 |
| L5 | 数据恢复 | 高 | 需人工 |

**太一权限**：可自动执行 L1-L3，L4-L5 需 SAYELF 批准

## 📈 效果评估

### 评估指标

| 指标 | 目标 | 计算方式 |
|------|------|---------|
| 自愈成功率 | ≥ 80% | 成功次数 / 总自愈次数 |
| 平均修复时间 | < 5 分钟 | 总耗时 / 自愈次数 |
| 人工干预率 | < 10% | 干预次数 / 发现问题次数 |
| 复发率 | < 20% | 同一问题复发次数 / 总问题数 |

### 周报生成

每周日 23:00 自动生成：
- 本周自愈统计
- 成功率趋势
- 高频问题分析
- 人工干预回顾
- 改进建议
