## 🎯 核心原则

> **Cron 任务神圣不可侵犯 · 未经太一允许禁止注释或删除**

**核心原则**:
1. **Cron 任务 = 系统生命线** - 每个任务都承载关键职责
2. **注释 = 切断生命线** - 临时调试不能成为永久禁用的理由
3. **太一审批 = 唯一授权** - 只有太一有权决定任务启停
4. **自动保护 > 人工记忆** - 用机制保障而非依赖人的记忆

---

## 🚨 问题根因分析

### 为什么 Cron 被注释？

| 原因 | 说明 | 频率 |
|------|------|------|
| **临时调试** | 开发时临时禁用，忘记恢复 | 高 |
| **脚本失效** | 依赖变化导致任务失败，被注释 | 中 |
| **资源冲突** | 多任务并发冲突，临时禁用 | 中 |
| **性能优化** | 减少负载临时禁用，未恢复 | 低 |
| **人员变更** | 原负责人离开，无人知晓 | 低 |

### 典型案例

```bash
# 原始配置 (正常运行)
0 * * * * /path/to/weather-forecast.sh

# 调试时临时注释
# 0 * * * * /path/to/weather-forecast.sh  ← 被遗忘

# 结果：任务永久失效，无人知晓
```

### 影响评估

| 影响 | 严重级别 | 说明 |
|------|---------|------|
| 数据采集中断 | P0 | 气象/交易数据缺失 |
| 监控失效 | P0 | 异常无法检测 |
| 报告缺失 | P1 | 日报/周报无法生成 |
| 策略失效 | P0 | 知几-E 无法交易 |
| 信任损失 | P1 | SAYELF 对系统失去信心 |

## 🛡️ 保护机制

### 机制 1: 权限锁定

**原则**: 仅太一可修改 crontab

**实现**:
```bash
# /etc/sudoers.d/crontab-protection
# 仅允许特定用户修改 crontab
Cmnd_Alias CRONTAB_CMD = /usr/bin/crontab
%taiyi-admin ALL=(ALL) CRONTAB_CMD
```

**执行**:
- 普通用户执行 `crontab -e` → 拒绝
- 太一执行 `crontab -e` → 允许
- 脚本自动修改 → 需太一授权

---

### 机制 2: 变更审计

**原则**: 所有修改必须记录

**实现**:
```bash
# crontab-pre-hook.sh (修改前自动执行)
BACKUP_FILE="/var/log/crontab-backup/$(date +%Y%m%d-%H%M%S).txt"
crontab -l > "$BACKUP_FILE"
echo "[$(date)] Backup created: $BACKUP_FILE" >> /var/log/crontab-changes.log
```

**审计日志**:
```
/var/log/crontab-changes.log
- 修改时间
- 修改内容 (diff)
- 修改人
- 修改原因
```

---

### 机制 3: 自动检测

**原则**: Task Orchestrator 每 30 分钟检查

**实现**:
```python
# task-orchestrator/cron_validator.py
def check_cron_integrity():
    current = get_current_crontab()
    baseline = get_baseline_crontab()
    
    disabled = find_disabled_tasks(current, baseline)
    if disabled:
        alert_taiyi(f"发现 {len(disabled)} 个任务被禁用")
        generate_audit_report(disabled)
```

**检测项**:
- 注释行数量变化
- 任务数量变化
- 执行频率变化
- 脚本路径变化

---

### 机制 4: 注释标记

**原则**: 临时注释必须标注原因和恢复时间

**格式**:
```bash
# [临时禁用] 2026-04-03 调试 weather-forecast.sh
# 原因：脚本依赖缺失
# 负责人：太一
# 恢复时间：2026-04-04 00:00
# 0 * * * * /path/to/weather-forecast.sh
```

**要求**:
- 必须标注 `[临时禁用]` 标记
- 必须说明原因
- 必须指定负责人
- 必须设定恢复时间 (≤7 天)

---

### 机制 5: 自动恢复

**原则**: 超期未恢复的任务自动激活

**实现**:
```python
# cron-auto-restore.py
def auto_restore_expired():
    disabled = find_disabled_tasks()
    for task in disabled:
        if task['restore_time'] < now():
            restore_task(task)
            notify_taiyi(f"自动恢复超期任务：{task['name']}")
```

**触发条件**:
- 超过恢复时间 7 天
- 负责人未申请延期
- 太一未明确禁止

---

### 机制 6: 备份保护

**原则**: 每日自动备份，保留 30 天

**实现**:
```bash
# /etc/cron.daily/backup-crontab
#!/bin/bash
BACKUP_DIR="/var/backups/crontab"
mkdir -p "$BACKUP_DIR"
crontab -l > "$BACKUP_DIR/crontab-$(date +%Y%m%d).txt"
find "$BACKUP_DIR" -mtime +30 -delete
```

**备份策略**:
- 每日 23:00 自动备份
- 保留 30 天
- 变更时立即备份
- 支持一键恢复

## 🔒 执行规则

### 铁律 (违反=严重事故)

1. **禁止无审批禁用** - 违者立即恢复 + 追责
2. **禁止无原因注释** - 违者视为恶意破坏
3. **禁止超期不恢复** - 违者自动恢复 + 通报
4. **禁止无备份修改** - 违者视为高风险操作
5. **禁止绕过 Task Orchestrator** - 违者触发告警

### 例外情况 (可先执行后补批)

| 情况 | 处理 | 补批时间 |
|------|------|---------|
| 安全漏洞 | 立即禁用 | 24 小时内 |
| 系统崩溃 | 立即禁用 | 24 小时内 |
| 数据泄露 | 立即禁用 | 24 小时内 |

## 📝 违规处理

### 违规级别

| 级别 | 行为 | 处理 |
|------|------|------|
| **轻微** | 未标注原因 | 警告 + 补标注 |
| **一般** | 超期未恢复 | 通报 + 强制恢复 |
| **严重** | 无审批禁用 | 追责 + 系统审查 |
| **极严重** | 恶意破坏 | 永久禁止修改权限 |

### 处理流程

```
检测违规 → 生成报告 → 太一审阅 → 执行处理 → 归档记录
```
