## 📋 核心原则

### 1. 价值优先原则

```
原则：
- 输出必须创造价值
- 废话 = 不输出
- 无价值任务不执行

应用：
- P0 任务必须执行
- P1 任务可跳过（如上下文紧张）
- P2 任务仅在心智充裕时执行
```

### 2. 负熵法则

```
原则：
- 任务执行减少混乱
- 不增加系统熵增
- 每次执行有明确产出

应用：
- 任务完成后立即归档
- 产出写入文件（不依赖记忆）
- 验证成果发送 Telegram
```

### 3. 时限承诺原则

```
原则：
- 任务有明确 deadline
- 超时立即告知
- 不拖延不积压

应用：
- P0 任务：5 分钟内完成
- P1 任务：15 分钟内完成
- P2 任务：30 分钟内完成
- 超时发送进度报告
```

### 4. 成果验证原则

```
原则：
- 任务完成必须验证
- 验证通过才关闭
- 验证失败要重试

应用：
- 文件生成 → 检查文件存在
- 数据入库 → 检查记录数
- 消息发送 → 检查 message_id
- 验证失败 → 自动重试 3 次
```

### 5. 通知透明原则

```
原则：
- 任务开始可通知
- 任务完成必通知
- 任务失败即告警

应用：
- P0 任务：开始 + 完成都通知
- P1 任务：仅完成通知
- P2 任务：仅失败通知
- 通知渠道：Telegram (7073481596)
```

### 6. 优先级调度原则

```
原则：
- P0 > P1 > P2
- 高优先级可打断低优先级
- 同优先级按时间排序

应用：
- P0 任务：立即执行
- P1 任务：排队执行
- P2 任务：空闲时执行
```

### 7. 资源节约原则

```
原则：
- 最小化 token 消耗
- 最小化 API 调用
- 最小化文件读写

应用：
- 批量处理 > 单个处理
- 缓存结果 > 重复计算
- 文件记忆 > 对话记忆
```

### 8. 自愈恢复原则

```
原则：
- 任务失败自动重试
- 连续失败告警
- 告警后人工介入

应用：
- 重试间隔：1 分钟
- 重试次数：3 次
- 连续失败：发送告警
```

### 9. 记忆固化原则

```
原则：
- 重要结论写入文件
- 不依赖对话记忆
- 每次 Session 可恢复

应用：
- 任务结果 → memory/YYYY-MM-DD.md
- 长期价值 → MEMORY.md
- 配置变更 → 对应配置文件
```

### 10. 人性克制原则

```
原则：
- 不表演不废话
- 直接执行不解释
- 有价值才输出

应用：
- 任务执行：直接做
- 成果交付：简洁清晰
- 遇到问题：直接说
```

## 📊 任务分级标准

### P0 任务（关键任务）

| 标准 | 说明 |
|------|------|
| **影响** | 影响核心业务 |
| **时效** | 错过无法弥补 |
| **频率** | 每日/每小时 |
| **例子** | 数据采集、交易信号、日报 |

### P1 任务（重要任务）

| 标准 | 说明 |
|------|------|
| **影响** | 影响次要业务 |
| **时效** | 错过可补救 |
| **频率** | 每日/每周 |
| **例子** | 热点搜索、健康检查、复盘 |

### P2 任务（一般任务）

| 标准 | 说明 |
|------|------|
| **影响** | 影响有限 |
| **时效** | 时间灵活 |
| **频率** | 每周/每月 |
| **例子** | 学习报告、文档整理、优化 |

## 📝 学习类任务调度原则

```
触发方式：
1. SAYELF 直接指令 → 立即执行
2. 定时学习任务 → 按 cron 执行
```

```
交付标准：
```

```
建议频率：
```

```
时间分配：
```

## 📋 任务执行流程

### 标准流程

```
1. 触发（cron/指令）
   ↓
2. 检查（资源/上下文）
   ↓
3. 执行（任务逻辑）
   ↓
4. 验证（成果检查）
   ↓
5. 通知（Telegram）
   ↓
6. 归档（文件/日志）
```

### 异常流程

```
1. 执行失败
   ↓
2. 自动重试（最多 3 次）
   ↓
3. 仍失败 → 发送告警
   ↓
4. 人工介入
   ↓
5. 问题解决 → 手动执行
   ↓
6. 记录原因 → 避免再犯
```
